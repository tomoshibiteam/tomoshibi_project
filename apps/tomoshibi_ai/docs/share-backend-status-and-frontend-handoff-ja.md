# TOMOSHIBI バックエンド状況・完成イメージ・フロント連携メモ（共有用）

**想定読者**: 経営・企画・デザイン・フロントエンド担当（技術に詳しくない方も含む）  
**更新方針**: 大きなリリースやAPI変更のたびに追記・更新してください。

---

## 1. この資料の目的

- **今のリポジトリが何をしているか**（バックエンド中心であること）を共通認識にする。
- **最終的に目指す体験**と、**いまどこまで来ているか**を一枚で把握する。
- **フロントエンド担当がつなぐときに必要な情報**（APIの考え方・注意点）をまとめる。

---

## 2. いまのリポジトリは「何のパーツ」か

| 言い方 | 内容 |
|--------|------|
| 一言 | **スマホアプリやWebの「画面」は含まず**、**サーバー側（バックエンド）**の実装が中心です。 |
| 役割 | ユーザーの外出セッション、周辺スポット、ルート案、相棒AIの返答、反応の記録、外出後の「思い出」化などを**APIとデータベースで処理**します。 |
| まだないもの | **ユーザーが触る本体のUI（フロントエンド）は別開発**です。計画書上も「フロントは未実装で最優先」とされています。 |

製品全体は「外出型AIコンパニオン」ですが、このコードベースはそのうち**頭脳と記録を担うサーバー部分**だと考えると分かりやすいです。

---

## 3. なぜこのプロダクト形か（背景・非技術者向け）

- **観光ガイドや旅行プランナーだけではない**: 地図アプリのように「場所を並べる」のではなく、**同じ相棒AIが続きを覚えて**、次の散歩や旅行の提案につなげることが目的です。
- **AIに「ウソの事実」を書かせない**: 店の存在・位置・営業時間などは**外部データや自前データ**から取り、AIは**言い方・共感・提案理由**に集中します。信頼と安全のための設計です。
- **初期は九州大学在学生の日常外出**を想定し、その後**壱岐などのエリアモード**へ同じ相棒を連れていく、という成長ストーリーが計画に書かれています。

---

## 4. 「完成形」に近い状態とは（100点イメージ）

マスタープランで定義されている「理想に近い状態」の要約です（全部が今日のバックエンドだけで完結するわけではありません）。

1. ユーザーが**現在地・時間・気分・興味**などを伝える。  
2. 周辺のスポット情報が**事実ベース**で集まる。  
3. **複数の歩き方・ルート候補**が出る。  
4. **相棒AI**が、関係性や過去の好みを踏まえて**なぜその候補か**を話す。  
5. 会話や「いいね／保存／スキップ／行った」などの**反応**ができる。  
6. それらが**記憶・関係性・次回の提案**に反映される。  
7. 外出が終わると**短い振り返り（思い出）**として残る。  
8. 将来は**地域の店・体験への自然な送客**までつながることを見据える。

**MVPであえてやらないこと**の例（計画より）: 本格予約の完結、複雑な最適ルート、3Dアバター、リアルタイム音声のみの体験、自治体向け大規模管理画面など。

## 5. 現状の進捗感（バックエンドで「もうできること」）

実装計画書（v2）の自己評価でも、**ゼロから作り直す段階ではない**とされています。いまのコードで、おおむね次の**一連の流れ（垂直スライス）**は通る設計になっています。

| ステップ | ユーザー視点のイメージ | バックエンドでの位置づけ |
|----------|------------------------|-------------------------|
| 1 | 今日の外出を始める | セッション作成 |
| 2 | 近くの候補を見たい | 周辺スポット取得（Places系 or 開発用モック） |
| 3 | 歩き方の案がほしい | ルート候補の生成 |
| 4 | 相棒が理由を話してくれる | AIによる提案文・会話の生成（設定によりモックも可） |
| 5 | 返事やボタン操作 | 会話・反応の記録 |
| 6 | 評価や保存 | フィードバック保存・記憶への反映の土台 |
| 7 | 今日はここまで | 外出完了・「思い出」化・関係性の更新 |
| 8 | 次回も覚えていてほしい | 過去の外出記憶を次の提案に載せる流れ |
| 9 | 外部サイトを開いた | リンククリックの計測（送客分析の入口） |

**キャラクター・見た目まわり**: 利用可能キャラの一覧、カスタマイズの取得・更新など、**APIレベルでは用意が進んでいます**。一方、計画書にある「パーツ解放専用API」「長期記憶をユーザーが編集・削除するAPI群」などは、**これから拡張される想定**です。

**一言で進捗を言うなら**:  
「**デモや結合テストに使えるバックエンドの芯**はあり、**プロダクト完成形のすべて**ではない（特にフロント・記憶UI・本番認証まわりはこれから）」が近いです。

## 6. 計画（v2）と比べて「まだ」または「これから厚くする」部分

非技術者向けに、**差分の意味**だけ書きます。

| テーマ | 計画上の方向性 | いまのコードベースとの関係 |
|--------|----------------|---------------------------|
| 初期ターゲット | 九大在学生の日常から始める | データ上は**モードやエリアID**で寄せられるが、**専用の「九大アプリ」画面はフロント側** |
| キャラクター | 複数キャラ・話し方の差はOK、**ジャンル縛りはしない** | 土台とAPIはある。**見た目のリッチ表現はフロント** |
| 長期記憶 | 一覧・編集・削除・同意など**ユーザーに見える記憶**へ | **構造化された「UserMemory」コレクションと専用APIは計画段階**。既存の要約・外出記憶は動く |
| 壱岐モード | 日常から旅へ**同じ相棒で接続** | エリアモード用の**シード（初期データ）の土台**はある。**導線の体験設計はフロントと仕様の共同** |
| フロント | 相棒感・見た目・記憶の**体験の中心** | **未実装（別リポジトリ／別チーム想定）** |

## 7. フロントエンド担当者への「伝えておくべきこと」

### 7.1 データの触り方（重要）

- **Firestore（クラウドのデータベース）をアプリから直接読み書きする想定ではありません。**  
  セキュリティルール上も、クライアントからの直接アクセスは閉じています。  
- **画面からは、公開されたHTTP API（Cloud Functions）を呼ぶ形**が基本です。

### 7.2 APIの呼び方（技術メモを最小限に）

- **HTTPの POST** で、**JSON形式**のリクエストボディを送ります（詳細な項目名は `functions/src/types/api.ts` に型としてまとまっています）。
- **関数名＝エンドポイント名**のイメージに近いです（例: `createGuideSession`）。デプロイ後の**実際のURL**は Firebase コンソールまたはデプロイ手順で確定します。
- **ヘルス確認用**に、サーバーが生きているか確認する簡単なAPIもあります。

### 7.3 認証について（プロダクト判断が必要）

- 現状のAPIは、リクエスト内の **`userId` を信頼する形**です。  
- **本番で「なりすまし防止」をどうするか**（Firebase Authentication と紐づける、IDトークン検証をサーバーで行う等）は、**フロント・バックエンド・インフラで事前に決める必要**があります。未決のまま公開すると危険です。

### 7.4 ブラウザやアプリから直接呼ぶとき

- **クロスオリジン（別ドメインのWebページから呼ぶ）**の場合、**CORS（サーバー側の許可設定）**が必要になることがあります。接続方式（同一ドメインのBFF経由にするか、Functions側でCORSを設定するか）は**早めに合意**してください。

### 7.5 開発・検証時の挙動

- **場所検索**や**AI**は、環境変数で **モック（ダミー）** に切り替え可能です。コストやキーなしでUIを繋ぎ込みたいときに便利です。
- **Java** が必要なのは、ローカルでデータベースエミュレータを動かすときなど（README参照）。

### 7.6 典型的な画面フローとAPIの対応（ざっくり）

| 画面のイメージ | 主に使うAPI（関数名） |
|----------------|----------------------|
| キャラを選ぶ・相棒の状態を見る | `getAvailableCharacters`, `getUserCompanionState` |
| 外出スタート | `createGuideSession` |
| ルート案と相棒の一文 | `suggestGuideRoute` |
| チャット・ボタン（いいね等） | `respondToCompanion`（必要に応じて `saveUserFeedback`） |
| 今日のおしまい | `completeJourney` |
| 見た目カスタマイズ | `getCharacterCustomization`, `updateCharacterCustomization` |
| 外部リンク（公式サイト等）を開く前の計測 | `trackOutboundClick`（返却されたURLへ遷移する想定） |

順序の目安: **セッション作成 → ルート提案 → 会話・反応 → 完了**です。

### 7.7 フロントが表示してよい「事実」と注意書き

- **地図上の位置・店名の存在・営業中か**などは、**APIが返したデータや外部取得結果を優先**し、AIの自由文だけを鵜呑みにしないUIが望ましいです（プロダクト原則）。
- AIの文には、**「最新の営業時間は公式で確認」**などの注記を併記できると安全です（バックエンド側でも同様のトーンが使われています）。

## 8. 関係者別の持ち帰りメッセージ

| 役割 | メッセージ |
|------|------------|
| **企画・事業** | バックエンドは「外出〜記録〜次回につなぐ芯」が動く段階。**体験の仕上げと検証の主戦場はフロント**。**九大→壱岐**のストーリーは仕様・画面で具体化が必要。 |
| **デザイン** | **相棒の一貫性・記憶の見せ方・行動導線**が差別化の中心。APIは「文と候補と状態」を返すので、**情報設計と感情設計**がフロントに集まる。 |
| **フロントエンド** | **POST JSON のAPI群**で結合。**認証・CORS・本番URL**は早めに決める。**userId の扱い**はセキュリティレビュー推奨。型の参照は `functions/src/types/api.ts`。 |
| **非技術者** | 「サーバー側の準備はデモ可能なところまで進んでいるが、アプリの画面は別。完成は両方そろって初めてユーザー体験になる」 |
